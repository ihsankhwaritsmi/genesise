# Omni-Disciplinary Knowledge Graph

A self-organizing personal knowledge base powered by AI agents in VS Code or the terminal. Inspired by Andrej Karpathy's "LLM Wiki" concept.

No vector databases. No graph databases. No servers. Just Markdown files, a rules engine, and an AI that builds and queries the graph for you.

## The Idea

You accumulate knowledge across a dozen domains — papers you read, meeting notes, code you studied, datasets you explored, ideas you had. It lives in scattered files and fades from memory.

This system turns all of it into a connected, queryable knowledge graph that an AI agent maintains and reasons over. Drop in a file. Tell the agent to process it. Come back later and ask a question — it retrieves the relevant context from your own graph before answering, not from its training data.

It works for anything: a research project, a hobby deep-dive, a startup, a novel you're writing, a personal reading log, competitive analysis, a D&D campaign. Any domain, any mix of domains.

## How It Works End to End

```
Clone repo
    ↓
Tell your agent: "Read cline_bootstrap.md and execute the instructions."
    ↓
Agent builds your workspace (folders, indexes, rules engine)
    ↓
Drop files into 01_raw_inputs/ — PDFs, notes, CSVs, code, transcripts, anything
    ↓
Tell your agent: "Process new data"
    ↓
Agent extracts concepts, writes structured nodes, links related ideas, flags contradictions
    ↓
Close your editor. Come back tomorrow. Start a new session.
    ↓
Ask anything: "Query the graph: [your question]"
    ↓
Agent reads your graph, reasons across it, falls back to the web if needed
    ↓
You get a grounded answer from your own accumulated knowledge
    ↓
Keep adding files. Keep asking. Keep building.
```

## Compatible Tools

Works with either:

| Tool | How it loads the rules |
|---|---|
| [Cline](https://marketplace.visualstudio.com/items?itemName=saoudrizwan.claude-dev) (VS Code extension) | reads `.clinerules` automatically on session start |
| [Claude Code](https://docs.anthropic.com/en/docs/claude-code) (CLI) | reads `CLAUDE.md` automatically on session start |

Both files are generated by the bootstrap with identical content — use whichever tool you prefer. You can switch between them freely since both work off the same graph files.

Any LLM API key works: Anthropic, OpenAI, Gemini, local via Ollama, etc.

## Prerequisites

- Cline (VS Code) **or** Claude Code (terminal) — whichever you prefer
- An LLM API key configured in your chosen tool

No installs, no databases, no configuration beyond what the bootstrap generates. Drop any file type — PDF, Word, Excel, image, CSV, code, plain text — and the agent reads it using tools already on your OS.

## Directory Structure

```
.
├── 01_raw_inputs/              # Drop your files here (never committed to git)
├── 02_nodes/                   # Generated knowledge graph nodes (never committed)
├── 03_indexes/                 # All index and config files (never committed)
│   ├── retrieval_protocol.md   ← full retrieval logic, loaded on demand
│   ├── input_manifest.md       ← source file tracking and change detection
│   ├── query_log.md            ← append-only history of every query run
│   ├── cluster_index.md        ← discipline-level pre-filter for large graphs
│   ├── node_registry.md        ← one row per node, primary retrieval index
│   ├── master_index.md         ← human-readable table of contents
│   └── source_config.md        ← external source settings
├── 04_synthesis/               # Cross-domain reports (never committed)
├── .clinerules                 # Rules engine for Cline (generated by bootstrap)
├── CLAUDE.md                   # Rules engine for Claude Code (identical to .clinerules)
├── .gitignore                  # Your data never gets committed
└── cline_bootstrap.md          # One-time setup directive
```

Everything under `01_raw_inputs/`, `02_nodes/`, `03_indexes/`, and `04_synthesis/` is gitignored — your personal knowledge never leaves your machine. `.clinerules` and `CLAUDE.md` are also gitignored since they are generated artifacts; `cline_bootstrap.md` is the committed source of truth.

## Commands

### `Process new data`
Scans `01_raw_inputs/` for new files, extracts structured knowledge into nodes, links related concepts, and updates all indexes.

### `Query the graph: [question]`
Standard query — reads up to 8 nodes. Retrieves from the local graph first, falls back to external sources if needed. Returns a confidence label, sources consulted, and any gaps.

```
Query the graph: What are the common failure modes across the architecture papers I uploaded?
Query the graph: What's the strongest counterargument to my current thesis?
Query the graph: What did I learn about fermentation that applies to this new recipe idea?
```

### `Query the graph [deep]: [question]`
Same but reads up to 15 nodes. Use for complex or cross-domain questions where breadth matters more than speed.

### `Synthesize across domains`
Finds nodes from different disciplines and writes a report applying the principles of one to the problems of another. Runs in deep mode. Adds backlinks to all source nodes.

### `Sync graph`
Run at the start of any session where `01_raw_inputs/` may have changed. Diffs the folder against the manifest and handles every case:
- **New** → processes and creates a node
- **Updated** → re-extracts, rewrites the node, updates connections and `last_verified`
- **Deleted** → removes the node, cleans up all references in other nodes
- **Unchanged** → skipped
- **Broken links** → scans all `[[WikiLinks]]` across all nodes and reports dead ones

### `Resolve contradiction: [node A] vs [node B]`
Reads both nodes, identifies the conflicting claims, searches for related evidence, and writes a reasoned resolution. Classifies as outdated, contextual, or genuinely open. Updates both nodes.

### `Search external: [topic]`
Forces a search against Wikipedia, ArXiv, and DuckDuckGo regardless of local graph state. Findings can be saved as nodes.

### `Compress node: [node name]`
Rewrites a verbose node body as a compact bullet-point list. YAML stays untouched. Updates `last_verified`.

## How Retrieval Works

Retrieval logic lives in `03_indexes/retrieval_protocol.md` and is only loaded when a query runs — keeping the rules file lean for every other session. No embeddings, no vector math — the LLM does the relevance judgment.

**Step 0 — HyDE**: Generates a hypothetical one-sentence summary of what the ideal answer node would say. Used as a second match signal alongside raw keywords.

**Step 1 — Cluster Pre-filter**: Reads `cluster_index.md` — one row per discipline. Narrows candidates to 1–3 disciplines before touching the full registry. At 300 nodes this means scoring ~20 rows instead of 300.

**Step 2 — Registry Scan**: Reads `node_registry.md` and scores each row against both the raw query and the HyDE summary. Assesses confidence: sufficient / partial / insufficient.

**Step 3 — Tiered Node Read**: Reads YAML block only (lines 1–30) for each candidate. Full file read only if confirmed relevant. Budget: 8 nodes normal, 15 nodes deep. Stale nodes flagged in the answer.

**Step 4 — Connection Traversal**: Follows `connections:` and `contradicts:` links up to 2 hops — only when the linked node's registry row also scores relevant.

**Step 5 — External Search**: Triggered when confidence is partial or insufficient. Hits Wikipedia, ArXiv, and DuckDuckGo via `curl`. No API keys needed.

**Step 6 — Answer + Log**: Returns confidence · answer · sources · gaps. Appends one row to `query_log.md`.

## Node Structure

```yaml
title: ""
type: ""          # research_paper | strategy | codebase | transcript | abstract_concept | dataset | synthesis | external
discipline: ""
tags: []          # broad categories
keywords: []      # specific terms, entities, acronyms — secondary retrieval signal
summary: ""       # one sentence, mandatory
assumptions: []
connections: []   # [[WikiLink]] to related nodes
contradicts: []   # [[WikiLink]] to conflicting nodes
source: ""
confidence: ""    # high | medium | low
date_added: ""    # set once, never changed
last_verified: "" # updated whenever the node is confirmed current
```
